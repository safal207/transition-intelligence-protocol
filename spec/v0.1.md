# Transition Intelligence Protocol v0.1

Status: Draft

## Purpose

TIP v0.1 defines the minimal record format for reasoning about a transition before action is taken.

The goal is simple:

```text
Do not jump from state directly to action.
```

A valid TIP record should pass through:

```text
State -> Tension -> Cause -> Transition -> Cooperation -> Action
```

## TIP Record

A TIP Record is a structured explanation of why an action is considered, what transition it may create, and whether cooperation remains stable after the action.

## Required fields

| Field | Meaning |
| --- | --- |
| id | Stable identifier for the record. |
| title | Human-readable title. |
| state | Current configuration of the situation. |
| tension | Pressure that makes the current state unstable or incomplete. |
| cause | Reason the transition exists or action may be considered. |
| transition | Movement from current state toward a possible next state. |
| cooperation | Stability check across participants. |
| action | Concrete next step. |
| status | Current lifecycle state of the record. |

## Status values

| Status | Meaning |
| --- | --- |
| draft | The record is still being formed. |
| clarified | State, tension, and cause are clear enough for review. |
| blocked | Action is not justified. |
| committed | Action is allowed as the next step. |
| reviewed | The result has been inspected after action. |

## Minimal invariants

A record is invalid if it contains an action but does not contain state, tension, cause, transition, and cooperation.

```text
state + tension + cause + transition + cooperation -> action
```

A record with `status = committed` must contain a non-empty, concrete `action.summary`. A commitment without a named action is not operationally meaningful.

## Field model

### State

The state describes the current situation before change.

It should include:

- actors;
- context;
- known facts;
- constraints.

### Tension

The tension describes why the state cannot simply remain as it is.

It may include:

- ambiguity;
- urgency;
- conflict;
- risk;
- opportunity;
- unmet need.

### Cause

The cause describes why the transition is meaningful or why action may be considered.

It should include:

- cause type;
- summary;
- supporting signals;
- confidence.

### Transition

The transition describes movement from the current state to a possible next state.

It should include:

- from state;
- to state;
- trigger;
- reversibility;
- expected consequence.

### Cooperation

The cooperation check asks whether the transition remains stable among important participants.

It should include:

- participants;
- shared upside;
- risk of unilateral breakage;
- stability level;
- recommendation.

### Action

The action describes the smallest justified next step.

It should be:

- proportional to the tension;
- based on cause;
- compatible with cooperation;
- reviewable later.

## Example

```json
{
  "id": "tip.example.startup_pivot",
  "title": "Startup pivot decision",
  "status": "committed",
  "state": {
    "summary": "The product has early interest but weak retention.",
    "actors": ["founder", "early users", "contributors"],
    "constraints": ["limited time", "unclear positioning"]
  },
  "tension": {
    "summary": "Users understand the idea but do not return consistently.",
    "kind": "unmet_need"
  },
  "cause": {
    "cause_type": "constraint",
    "summary": "The current product solves a broad problem, not an urgent narrow pain.",
    "supporting_signals": ["weak retention", "unclear user habit"],
    "basis": "Observed usage pattern",
    "confidence": 0.72
  },
  "transition": {
    "from": "broad productivity product",
    "to": "narrow decision-support protocol",
    "trigger": "retention weakness",
    "reversibility": "medium",
    "expected_consequence": "clearer value proposition"
  },
  "cooperation": {
    "participants": [
      {"name": "founder", "goal": "focus", "risk": "overbuilding"},
      {"name": "early users", "goal": "clear benefit", "risk": "confusion"}
    ],
    "shared_upside": "Sharper product category and easier validation.",
    "defection_risk": "medium",
    "stability": "fragile",
    "recommendation": "clarify"
  },
  "action": {
    "summary": "Run one narrow pilot before changing the whole product.",
    "review_after": "pilot completion"
  }
}
```

## Version goal

v0.1 is intentionally small. It exists to make one transition inspectable.

Future versions can add scoring, canonical transition states, replay, CLI tooling, and integration with agent traces.
